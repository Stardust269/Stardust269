## 五、时间窗口与「最近一份」口径

### 5.1 「最近一份」征信报告

`latest_report_rn = 1` 的选取逻辑（Step 5）：

1. **优先**：观察日 `days_dt` **当日及之前**的最新报告（`days_dt_zx <= days_dt`）
2. **兜底**：若观察日前无报告，则取观察日后 **60 天内**最新报告（用于标签窗内兜底）

### 5.2 近一月 / 近 6 月 / 近一年

以 `days_dt` 为锚点，向前回看：

| 窗口 | 条件 | 聚合方式 |
|------|------|----------|
| 近 1 月 | `days_dt - 30 < days_dt_zx <= days_dt` | 多份报告 **均值**（`avg`） |
| 近 6 月 | `days_dt - 180 < days_dt_zx <= days_dt` | 多份报告 **均值** |
| 近 1 年 | `days_dt - 365 < days_dt_zx <= days_dt` | 多份报告 **均值** |

循环贷类（账户数、余额、授信、利用率）：最近一份取 `max(latest_report_rn=1, ...)`，历史窗口取 `avg(flg_win_*=1, ...)`。

### 5.3 征信关联总窗

`days_dt` 前推 **365 天** ～ 后推 **60 天**（与标签观察窗一致）。

### 9.1 特征表 0kb / 0 行（常见原因与处理）

数据地图显示 `jcr_credit_feature_20260715` 为 **0kb、无数据** 时，按下面顺序排查：

| 优先级 | 原因 | 现象 | 处理 |
|--------|------|------|------|
| **①** | `create table if not exists` 曾**空跑**过 | 表结构在、0 行；重跑 Step6 仍 0 行 | **先 `drop table`** 再重跑 Step 5～7（新版 SQL 已改为 drop+create） |
| **②** | 样本表 `jcr_pril_bal_info_20260715` 为空 | Part5/6 均为 0 | 确认同事 `yye_pril_bal_info_20260715_1` 已就绪；必要时 `drop_all_jcr_tables_20260715.sql` 后重跑 |
| **③** | 只跑了 Part3~4，未跑 Part5~6 | 中间表有数，feature 表 0 行 | 补跑 Part 5 → 6 → 7 |
| **④** | 旧版 Part5 的 `WHERE` 滤掉全部用户 | 样本有数，Part5=0 | 征信报告全落在 `days_dt±窗口` 外时，旧逻辑会把用户整行删掉；**已修复**：时间窗改到 `JOIN ON` |
| **⑤** | `uuid` ≠ `id_unqp` | 样本有数，关联率极低，特征全 null | 执行 Step8-②；若匹配率极低需确认样本表关联键 |

**建议操作（一次性）**：

```sql
-- 1. 若曾空跑，先删空表
drop table if exists lj_iceberg.ai_decision_dev.jcr_credit_feature_20260715;
drop table if exists lj_iceberg.ai_decision_dev.jcr_credit_feature_label_full_20260715;
drop table if exists lj_iceberg.ai_decision_dev.jcr_credit_report_with_sample_20260715;

-- 2. 确认样本表有数据（应 > 0）
select count(1) from lj_iceberg.ai_decision_dev.jcr_pril_bal_info_20260715;

-- 3. 重跑 Step 5 → 6 → 7（Step 1～4 可跳过）
```

完整排查 SQL 见 `sql/run_all_20260715.sql` 末尾 **Part 9** 核验块。

1. **数据量极大**：Step 1～3 为全量征信用户加工，可考虑先缩样本用户再关联征信，降低中间表体积
